# Routes

All routes are registered on the FastAPI app in `main.py`. Every path is prefixed with `/api/`.

Interactive documentation (auto-generated by FastAPI) is available at `/docs` when the server is running.

---

## Plants — `plants.py`

### `POST /api/plants/sync`

Receives the current plant list from the frontend and upserts it into `plants_raw`. Triggers a pipeline run (ingestion + transform + aggregation) with the incoming data.

**Request body**

```json
{
  "plants": [
    {
      "id": "string",
      "name": "string",
      "location": "string | null",
      "interval": 7,
      "lastWatered": "2024-03-15T10:00:00Z | null",
      "isDead": false
    }
  ]
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | string | yes | Client-generated ID (typically `Date.now()`) |
| `name` | string | yes | |
| `location` | string | no | |
| `interval` | integer | yes | Watering interval in days |
| `lastWatered` | string | no | ISO 8601 timestamp |
| `isDead` | boolean | no | Defaults to `false` |

**Response**

```json
{
  "ok": true,
  "staged": 3
}
```

`staged` is the number of plant rows upserted.

---

### `GET /api/plants`

Returns all plants from `plants_raw`, each joined with that plant's most recent health metrics row.

**Response** — array of plant objects

```json
[
  {
    "id": "1710500000000",
    "name": "Monstera",
    "location": "Living Room",
    "interval": 7,
    "last_watered": "2024-03-14T09:00:00+00:00",
    "is_dead": 0,
    "synced_at": "2024-03-15T10:00:00+00:00",
    "metrics": {
      "health_score": 0.8250,
      "compliance_rate": 1.0,
      "avg_days_overdue": 0.42
    }
  }
]
```

`metrics` is `null` if the plant has no computed health metrics yet (i.e. no completed care events have been processed by the pipeline).

## Events — `events.py`

### `POST /api/events/sync`

Receives care events from the frontend and upserts them into `events_raw`. Triggers a pipeline run with the incoming events.

**Request body**

```json
{
  "events": [
    {
      "id": "string",
      "plantId": "string",
      "eventType": "watering",
      "scheduled": "2024-03-15T09:00:00Z",
      "completed": "2024-03-15T10:30:00Z | null",
      "feedback": "happy | sad | overwatered | null"
    }
  ]
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | string | yes | Client-generated event ID |
| `plantId` | string | yes | Must match an existing `plants_raw.id` |
| `eventType` | string | yes | e.g. `"watering"` |
| `scheduled` | string | yes | ISO 8601 — when watering was due |
| `completed` | string | no | ISO 8601 — when it was actually done |
| `feedback` | string | no | `"happy"`, `"sad"`, or `"overwatered"` |

**Response**

```json
{
  "ok": true,
  "staged": 12
}
```

`staged` is the number of event rows upserted.

## Analytics — `analytics.py`

### `GET /api/analytics/summary`

Aggregate stats across all plants and events.

**Response**

```json
{
  "total_plants": 5,
  "total_events": 47,
  "completed_events": 43,
  "overdue_events": 4,
  "compliance_rate": 0.9149,
  "avg_health_score": 0.7832
}
```

| Field | Notes |
|-------|-------|
| `total_plants` | Count of rows in `plants_raw` |
| `total_events` | Count of rows in `care_events` |
| `completed_events` | Events with a non-NULL `completed` timestamp |
| `overdue_events` | Events where `days_overdue > 1` |
| `compliance_rate` | `completed_events / total_events`, or 0 if no events |
| `avg_health_score` | Average of the most recent `health_score` per plant; `null` if no metrics exist |

---

### `GET /api/analytics/trends`

Daily breakdown of care events for the last 28 days, ordered most-recent first.

**Response** — array of up to 28 day objects

```json
[
  {
    "day": "2024-03-15",
    "events": 3,
    "completed": 3
  },
  {
    "day": "2024-03-14",
    "events": 2,
    "completed": 1
  }
]
```

`events` is the total number of care events scheduled on that day. `completed` is how many were actually completed.

---

### `POST /api/analytics/run-pipeline`

Manually triggers the ETL pipeline (transform + aggregation only — no new data is ingested since no plants or events are passed). Useful for forcing a recompute after bulk data changes.

**Request body** — none

**Response** — same shape as the runner result dict

```json
{
  "plants_staged": 0,
  "events_staged": 0,
  "events_transformed": 2,
  "metrics_computed": 5,
  "status": "success",
  "error": null
}
```

---

### `GET /api/analytics/export`

Full data dump — all three data tables as JSON arrays. Useful for backup, debugging, or migrating to a different database.

**Response**

```json
{
  "plants": [ ...all rows from plants_raw... ],
  "events": [ ...all rows from care_events... ],
  "metrics": [ ...all rows from plant_health_metrics... ]
}
```

Note: this returns `care_events` (the transformed table), not `events_raw`.

---

### `GET /api/analytics/pipeline-runs`

The last 50 ETL audit log entries, ordered most-recent first.

**Response** — array of up to 50 run objects

```json
[
  {
    "id": 42,
    "started_at": "2024-03-15T10:05:00+00:00",
    "finished_at": "2024-03-15T10:05:00.123+00:00",
    "status": "success",
    "plants_staged": 0,
    "events_staged": 0,
    "events_transformed": 0,
    "metrics_computed": 5,
    "error": null
  }
]
```

A `status` of `"running"` with a NULL `finished_at` means the pipeline is currently in progress (or crashed before it could update the row).
