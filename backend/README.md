# backend/

A FastAPI server that receives plant and watering data from the frontend, runs a 3-layer ETL pipeline every 5 minutes, and exposes analytics endpoints. It also serves `frontend/index.html` as a catch-all so the entire app runs from a single URL.

---

## Files

**`main.py`**
Entry point. Registers all three route groups under `/api/`, adds CORS middleware (allows all origins so the app works from any deployment URL), and on startup: calls `init_db()` to create tables if they don't exist, then starts APScheduler to call `run_pipeline()` every 5 minutes. Also mounts two routes — `/` and `/{full_path}` — that serve `frontend/index.html`, which is how a single Render service hosts both the API and the UI.

**`db.py`**
Opens a SQLite connection in WAL (Write-Ahead Logging) mode, which allows concurrent reads during a write. `row_factory = sqlite3.Row` means every query result is accessible by column name. `init_db()` creates all 6 tables on first run using `CREATE TABLE IF NOT EXISTS`, so it's safe to call on every startup.

**`models.py`**
Pydantic models that validate the JSON body of incoming sync requests. `PlantIn` and `EventIn` map frontend camelCase field names (`lastWatered`, `plantId`, `isDead`) to the Python snake_case expected by the pipeline. FastAPI uses these automatically — a malformed request returns a 422 before it touches any database code.

## Database tables

**`plants_raw`** — Staging table. Every plant synced from the frontend lands here. The primary key is the frontend-generated UUID, so syncing the same plant twice is an upsert (`ON CONFLICT DO UPDATE`), not a duplicate insert.

| Column | Type | Description |
|---|---|---|
| id | TEXT PK | UUID generated by the frontend |
| name | TEXT | Plant name |
| location | TEXT | Optional location label |
| interval | INTEGER | Current watering interval in days |
| last_watered | TEXT | ISO timestamp of last watering |
| is_dead | INTEGER | 0 = alive, 1 = dead |
| synced_at | TEXT | When this record was last received |

**`events_raw`** — Staging table for care events. Same upsert behaviour — if an event is synced again (e.g. the completion timestamp was added), only `completed` and `feedback` are updated.

| Column | Type | Description |
|---|---|---|
| id | TEXT PK | UUID generated by the frontend |
| plant_id | TEXT | References plants_raw.id |
| event_type | TEXT | e.g. `water` |
| scheduled | TEXT | ISO datetime the watering was due |
| completed | TEXT | ISO datetime it was actually done (null if skipped) |
| feedback | TEXT | `happy`, `sad`, or `overwatered` |
| synced_at | TEXT | When this record was last received |

**`care_events`** — Transformed table. Only completed events appear here, enriched with computed fields by `transform.py`.

| Column | Type | Description |
|---|---|---|
| id | TEXT PK | Same as events_raw.id |
| days_overdue | REAL | `(completed - scheduled)` in days. Negative = early, positive = late |
| was_on_time | INTEGER | 1 if days_overdue ≤ 1, else 0 |

**`plant_health_metrics`** — One row per plant per pipeline run. The latest row for each plant is used by the summary endpoint.

| Column | Type | Description |
|---|---|---|
| health_score | REAL | 0–1 composite score |
| compliance_rate | REAL | Fraction of events completed |
| avg_days_overdue | REAL | Average lateness across all events |
| total_events / completed_events | INTEGER | Raw counts |

**`weather_snapshots`** — Reserved for future use. The schema exists but nothing writes to it yet; weather data currently lives only in the frontend.

**`pipeline_runs`** — Audit log. Every pipeline execution — scheduled or manually triggered — writes one row here with counts of records processed at each layer and any error message.

## API endpoints

| Method | Path | What it does |
|---|---|---|
| POST | `/api/plants/sync` | Upserts plants from the frontend into `plants_raw`, then triggers a pipeline run |
| GET | `/api/plants` | Returns all plants with their latest health metrics joined in |
| POST | `/api/events/sync` | Upserts care events into `events_raw`, then triggers a pipeline run |
| GET | `/api/analytics/summary` | Total plants, events, compliance rate, average health score across all plants |
| GET | `/api/analytics/trends` | Daily event counts and completion counts for the last 28 days |
| POST | `/api/analytics/run-pipeline` | Manually triggers the full ETL pipeline |
| GET | `/api/analytics/export` | Full dump of plants, care_events, and health metrics as JSON |
| GET | `/api/analytics/pipeline-runs` | Last 50 ETL audit log entries |

See [routes/README.md](routes/README.md) for full request/response shapes.

## ETL pipeline

See [pipelines/README.md](pipelines/README.md) for the full breakdown. In short: ingestion stages raw data, transform enriches completed events with timing metrics, aggregation rolls up a health score per plant. Every run is logged to `pipeline_runs`.
